# OPERATOR 13 · STUDIO — theory & field notes

Single-file HTML. No runtime dependencies (QR encoder already vendored). Same Worker
protocol as OP-12, now tolerant of the *old* deployed Worker.

---

## 1 · PURPOSE

OPERATOR is a film instrument, not a videoconference. OP-13 makes that literal:
the camera is a **capture device** that films what is in front of you into the
branching film tree, and the mesh is a **crew channel** that must never gate
whether someone can work. Two people in a room with one camera between them can
still make a film together.

Three ruling ideas:

1. **Capture is authorship.** A photo or a held clip enters the tree as frames,
   exactly like a drawn frame or an AI-composed sequence. Same codec (128×96
   Sobel edge grid), same provenance system, same branch grammar.
2. **Presence before pixels.** Knowing someone is IN ROOM is separate from
   having their video LINKED. The old build conflated them, so a camera failure
   read as "nobody joined."
3. **Degrade, never gate.** No camera → you still join, draw, write, prompt,
   receive. Old Worker without roster → you still pair as two peers. Every
   failure has a floor.

## 2 · ENTITIES

| Entity | What it is | Bound |
|---|---|---|
| `capsurf` | Fullscreen capture surface: live edge-preview, shutter, lens flip, fps select, no-feed fallback | one instance |
| `capBuf` | Frames accumulating during a held shutter | `REC_MAX = 32`, further clamped by `capHeadroom = NODE_MAX − treeSize − 1` |
| `roomRoster` | Worker's authoritative list of peers IN ROOM | `MAX_OPERATOR_PEERS = 6` (Worker) |
| `conns` | Per-peer RTCPeerConnections (mesh Part B) | ≤ roster size |

| `room.legacyPolls` | Throttle counter for legacy pairing kicks | resets on link |
| crew rows | Rendered roster entries in C · CREW | `CREW_ROWS_MAX = 8` |
| PROV kinds | + `capture:'CAPTURE'`, `clip:'CLIP'` | closed map |

## 3 · OPERATIONS

**CAPTURE** (A · AUTHOR, first group; also `op-cap` on the timeline)
- *tap* shutter → one frame → `appendSequence([g],'capture')`
- *hold* ≥300 ms → record at 4/8/12 fps → release → `appendSequence(frames,'clip')`
- *flip* → `setFacing()` swaps the video track **in place** (kept audio tracks,
  no renegotiation) — during a call this deliberately re-aims what the crew
  sees. The camera is a location scout, not a mirror.
- *no feed* → surface stays usable: Load photo / video routes through the same
  `gridFromSource → appendSequence` path.

**JOIN, camera-optional**
- All three call sites (host go-live, paste-join, join-go) run
  `startCameraOptional`: camera success and camera denial both proceed to
  `joinRoom`. Denial toasts "No camera — continuing without it."
- Connect button is never disabled while OFFLINE.

**PRESENCE**
- `refreshPresence` renders two truths separately: N IN ROOM (roster) and
  LINKED (open data channels). A guest with a broken camera shows as
  `JOINING…` instead of not existing.
- `renderCrew` lists each peer: id tail, role, LINKED / IN ROOM.

**LEGACY PAIRING** (`legacyRosterKick`)
- If polls return no `roster` field (old deployed Worker): never drop conns on
  roster silence; when peers ≥ 2 and nothing is linked, broadcast a `restart`
  on polls 1, 5, 9…; the deterministic impolite side (lower peerId) offers.
  Converges to the old 2-peer behavior. Stops the moment a roster appears or a
  link opens. Logged as `LEGACY`.

**RECOVERY** — `recoverConnection` / `tryIceRestart` route through
`meshRestartAll()`: per-conn, impolite side does `createOffer({iceRestart:true})`
targeted via `to`; polite side triggers one broadcast restart. Manual-SDP path
keeps `pc.restartIce` fallback.

## 4 · INVARIANTS

- Single file; CDN-free at runtime (QR already vendored — see §7).
- Absolute black/white + named accents only; 3px borders; `6px 6px 0` shadows;
  **no border-radius** on rectangular containers; no emoji glyphs — every mark
  is drawn vector/CSS.
- All loops named-bounded (Power-of-10): `REC_MAX`, `CREW_ROWS_MAX`,
  `NODE_MAX`, `FLOW_MAX`, `MAX_MSGS`, `MAX_OPERATOR_PEERS`.
- Capture never auto-stops the camera on close (consistent with the app's
  standing no-surprise-camera-off behavior).
- `appendSequence` is the single doorway into the tree for all captured media.
- Worker protocol unchanged; client is a strict superset of OP-12's.

## 5 · FAILURE MODES → FLOORS

| Failure | Floor |
|---|---|
| Camera denied / absent | Join proceeds; AUTHOR still has Draw / Write / Prompt / Shape; capture surface offers file-load |
| Old Worker (no roster) | 2-peer legacy pairing; `LEGACY` in log; no conn-drop storms |
| Guest joins, WebRTC fails | Host still sees them IN ROOM / JOINING… in crew list |
| Room full (Worker cap) | Copy tells guest to ask HOST to end + restart |
| Tree near `NODE_MAX` | `capHeadroom` clamps clip length; shutter refuses past ceiling |
| `getUserMedia` mid-call track death | flip/retry path re-acquires without renegotiation |

## 6 · TEST LADDER

MESH.md steps 1–6 still apply (solo → 2 → 3 peers, refresh-rejoin, host-drop,
cap). OP-13 adds:

7. **Guest denies camera** → guest lands in room; host's CREW shows them
   JOINING… then LINKED; guest can draw and send frames? (send requires no
   camera — control channel only).
8. **Legacy Worker** → point client at pre-mesh Worker; two peers converge
   within ≤ 5 polls; log shows `LEGACY`; no repeated conn teardown.
9. **Capture tap** → one `CAPTURE` node appended to current branch.
10. **Capture hold** → clip of ≤ 32 frames as `CLIP`; count overlay matches.
11. **Flip during call** → remote view re-aims; audio uninterrupted.
12. **No-feed fallback** → block camera, open capture, load a video file,
    confirm frames enter tree.

## 7 · QR NOTE

OP-12 shipped with the encoder **already vendored**
(`qrcode-generator@1.4.4`, Arase, MIT — see the banner in the file). OP-13
carried that block over intact, so **no build step is needed**: invite QR works
out of the box. `build-qr.js` only matters if you ever restore the
`/* QR-VENDOR-PLACEHOLDER */` block; it would refuse to double-inline as-is.

## 8 · DEPLOYMENT

- **Client:** publish `operator-studio-13.html` (e.g. to
  `hartswf0.github.io/prompt-language/`).
- **Worker:** the included `worker.js` is unchanged from your upload and
  already carries mesh Part A (roster, `to` routing, `MAX_OPERATOR_PEERS=6`,
  `MAX_MSGS=256`). **Deploying it is required for >2 peers.** Until then the
  client self-detects the old Worker and runs the 2-peer legacy floor.

## 9 · CHANGE SCENARIOS (growth)

- **Past ~6 peers:** full mesh is O(N²) video; the honest next step is an SFU
  (or: mesh for control, star for video through the host).
- **Per-peer attribution:** `onControl` already threads `fromId` — ink color /
  cursor ghosts per operator is a rendering change, not a protocol change.
- **Capture onion-skin:** ghost the previous frame under the live preview at
  ~30% for stop-motion registration; one compositing line in
  `paintCapPreview`.
- **Facing memory:** persist last-used lens per room role (scout rooms tend
  environment-facing).
- **Godot bridge:** captured `CLIP` sequences already serialize like any
  branch; the glTF+sidecar exporter can treat them as texture flipbooks.
